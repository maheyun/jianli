## 一、SQL & 数据库（必考，面试第一关）

### Q1. 五表 JOIN 性能排查

**"五表 JOIN 查老用户 RFM 数据，如果发现 SQL 跑了 30 秒还没出结果，你怎么排查？"**

这道题考察的是你项目二中写过的多表关联。面试官想听的不是语法，是你对 **执行计划、索引、数据量** 的感知。

<details>
<summary>点击查看回答要点</summary>

- 先 `EXPLAIN` 看执行计划，检查有没有全表扫描（type=ALL）
- 看 JOIN 顺序：是不是先 JOIN 大表再 JOIN 小表导致临时表膨胀
- 看 WHERE 条件有没有走到索引：`order_status` 这列如果是散列分布（只有 paid/unpaid/shipped 几种值），B-tree 索引作用有限
- 看关联列 `user_id` 有没有索引，这条一般是最关键的
- 如果数据量大到 SQL 层面已无法优化，考虑是否应该分批查询后在 pandas 里 merge

</details>

### Q2. LIKE 与索引的关系

**"你在学习指南里写了 `LIKE 'abc%'` 能走索引、`LIKE '%abc%'` 不能——为什么？这和 B-tree 的物理结构有什么关系？"**

这正是你 `知识点` 文件里记过的问题。

<details>
<summary>点击查看回答要点</summary>

- B-tree 索引是按前缀排序存储的。`LIKE 'abc%'` 可以定位到 'abc' 开头的叶子节点然后顺序扫描，所以能利用索引
- `LIKE '%abc%'` 的 'abc' 可能在字符串中间的任意位置，索引的前缀排序特性失效，必须全表扫描
- 全模糊匹配的正确方案是全文索引（FULLTEXT）+ `MATCH ... AGAINST`

</details>

### Q3. CASE WHEN 与 WHERE 的区别

**"你的项目二里写了 `COUNT(DISTINCT CASE WHEN p.category_id = 2 THEN o.order_id END)`——这个写法和 `WHERE p.category_id = 2` 有什么区别？"**

<details>
<summary>点击查看回答要点</summary>

- 加 WHERE 会过滤掉整行，丢失那些没有买过服装订单的用户行
- 用 `CASE WHEN` 在聚合函数内部做条件判断，保留所有用户行，只是对不满足条件的不计数
- 这正是你项目二要做左联但又不丢失 `仅袜子` 用户数据的原因

</details>

### Q4. SQL 分箱（NTILE）

**"关于 `pd.qcut` 报 ValueError 的问题（你学习指南里记录的 `safe_qcut`）——面试官可能会问：如果不用 Python，纯 SQL 怎么做用户 RFM 分箱？"**

<details>
<summary>点击查看回答要点</summary>

- 用 `NTILE(5) OVER (ORDER BY recency)` 等分，`NTILE` 的底部分箱逻辑和 `pd.qcut` 不完全一样但用途相同
- 如果 recency 有大量重复值，`NTILE` 不会报错——它用循环赛分配（round-robin）而不是严格边界切割
- 对比：`NTILE` 保证每箱行数相等但可能把相同值的行分到不同箱；`pd.qcut` 严格按值切分

</details>

## 二、Python / pandas（代码功底）

### Q5. np.select 为什么快？

**"你在学习指南里写 `np.select` 替代 `.apply(axis=1)` 快 10~50 倍。请现场解释原理：为什么快了这么多？"**

<details>
<summary>点击查看回答要点</summary>

- `.apply(axis=1)` 对每一行调用一次 Python 函数——20,000 行 = 20,000 次函数调用的开销（每次创建/销毁栈帧）
- `np.select` 在 numpy 底层 C 语言实现，一次性对所有行做向量化比较，没有 Python 函数调用
- 比较的不是判断逻辑本身，是"调用 Python 函数"这件事的开销

</details>

### Q6. pivot_table 的 OOM 问题

**"你项目一的漏斗分析用了 `pivot_table`，但如果数据量是 1000 万行，pivot 可能 OOM——有哪些替代方案？"**

<details>
<summary>点击查看回答要点</summary>

- SQL 里做聚合（GROUP BY），只拉汇总结果到 Python
- 分批读取（`pd.read_sql(chunksize=...)`）+ 逐步聚合
- 如果 SQL 能完成计算就不要把原始数据全拉到 Python 内存

</details>

### Q7. 逐点差分 vs 多项式回归

**"你在项目三里写了一个 `calculate_marginal_roi` 函数（逐点差分），后来陈立让你换成多项式回归。如果面试官追问：逐点差分算出来的边际 ROI 有什么问题？"**

<details>
<summary>点击查看回答要点</summary>

- 逐点差分 `(rev[i] - rev[i-1]) / (spend[i] - spend[i-1])` 对数据中的微小波动极度敏感——分母接近 0 时结果爆炸
- 多项式回归通过拟合函数曲线来估计边际斜率，能平滑掉噪声
- 但多项式回归要警惕过拟合——项目三用了二次项（2 次），没上高次项，这是合理的
- 更严谨的做法可以考虑对数变换或饱和增长曲线

</details>

## 三、统计学 & AB 测试（面试重点，杀伤力强）

### Q8. 为什么卡方检验 + z 检验都要用？

**"你项目二里 AB 测试同时用了卡方检验和双样本比例 z 检验——面试官一定会追问：为什么两个都要用，而不是只用一个？"**

这是你学习指南里提到陈立让你做的事——这是面试官最喜欢深挖的点。

<details>
<summary>点击查看回答要点</summary>

- **卡方检验**回答"四组之间有没有差别"——这是一个整体检验（omnibus test）。拒绝 H₀ 只能说明"至少有一组不同"，不知道是哪组
- **z 检验**回答了"哪组比哪组好"——两两比较（pairwise comparison）。比如对照组 vs 测试组C，哪组转化率显著更高
- 相当于医生先做全面体检（卡方）告诉你"有问题"，再做针对性检查（z）告诉你"问题在哪"

</details>

---

### Q9. 多重比较的陷阱

**"两两比较做多组 z 检验，你有什么需要注意的？"**

<details>
<summary>点击查看回答要点</summary>

- 多重比较问题（multiple comparisons）——做的比较越多，犯 Type I Error（假阳性）的概率越大
- 解决方法：Bonferroni 校正（把显著性水平 α 除以比较次数），或者 Tukey HSD
- 如果面试官追问"你的代码里做了吗"，诚实回答：当时用简单方法，这是可以继续改进的点

</details>

### Q10. 样本量与功效分析

**"AB 测试中，你的实验跑了多久？样本量怎么确定的？如果面试官说'转化率从 5% 提升到 5.3%，1000 个样本够不够得出显著结论'——你怎么回答？"**

<details>
<summary>点击查看回答要点</summary>

- 这个回答需要功效分析（power analysis）——样本量不足导致即使有真实效果也检测不出来（Type II Error）
- 可以用 `statsmodels` 的 `power_proportions_2indep` 计算所需最小样本量
- 你的项目二里模拟数据用的是 8000+ 用户，但真实场景需要先做 power analysis 再跑实验

</details>

### Q11. G 维度分母加 1 的问题

**"你在项目二的 G 维度设计里用了 `category_2_orders / (category_1_orders + 1)`。面试官如果追问：为什么分母加 1？这有什么统计学问题？"**

<details>
<summary>点击查看回答要点</summary>

- 加 1 是防止分母为 0（用户从来没买过袜子品类）。这是工程上的防御性写法
- 但这是对缺失数据的"假值填充"（imputation with arbitrary constant），会引入偏差
- 更严谨的做法是用贝叶斯平滑（bayesian shrinkage）——给低频用户一个合理的先验估计，而不是硬加 1
- 你可以说：当时数据量下这个偏差影响不大，但面试时展示出你知道有更好的方法

</details>

## 四、业务分析能力（最能区分你和背答案的人）

### Q12. 四平台预算分配的决策逻辑

**"你们有四个平台：天猫 ROI 高、得物 ROI 44 倍、抖音拉新好但 ROI 只有 16。如果吴总监问你'预算应该优先往谁倾斜'，你怎么回答——不要只说答案，讲你的分析逻辑。"**

<details>
<summary>点击查看回答要点</summary>

- 第一步不是直接看 ROI 数字——得物 44× 是不是因为样本小？
- 第二步看边际 ROI：得物投 10 万时 ROI=44，投 20 万时 ROI 是否骤降到 20？这才是决策关键
- 第三步考虑平台角色：得物是"高价值种草地"（转化好但用户群小），抖音是"高效拉新场"（量大但 ROI 低），不会 all-in 一个而是结构调整
- 最终决策框架：综合 ROI(60%) + 边际 ROI(40%) 权重分配——这是你项目三实际做过的

</details>

### Q13. GMV 骤降排查

**"如果 GMV 突然下降 20%，你怎么排查？"**

这道题就是学习指南 7.4 列出的经典面试题。

<details>
<summary>点击查看回答要点</summary>

逐层拆解，向下钻取（Drill-down）：

1. **是不是全渠道都跌了？** 还是某个渠道暴跌拉低了总和？→ 分渠道看
2. **是流量掉了还是转化掉了？** `GMV = 流量 × 转化率 × 客单价` → 分别拆
3. **如果是流量掉了：是不是某个来源断流了？** 付费 vs 自然流量
4. **如果是转化掉了：跌在漏斗哪个环节？** 浏览→加购→下单→支付
5. **是不是昨天做了什么？** 大促后回落？会员日？平台改算法？
6. **是不是数据本身有问题？** 参考孙倩的 `异常备注 Sheet`——哪个平台数据没入库

</details>

### Q14. ROI 口径陷阱

**"你说'得物 ROI 44 倍，抖音 16 倍'。面试官可能会笑一下然后问：这数字怎么算的？抖音的 ROI 算法和得物算的是同一个口径吗？"**

这是个陷阱题。面试官在测试你是否知道数据分析中最常见的坑——**口径不统一**。

<details>
<summary>点击查看回答要点</summary>

- 项目三中郑浩告诉过你：抖音花费是含税的，天猫不含税
- 得物 9 月之前数据口径不一样
- 所以直接拿 44× 和 16× 比较本身就有问题
- 正确的做法是先做口径对齐（统一换算为不含税花费），再比较
- 这就是你学习指南里写的"数据在哪、格式是什么、有没有坑"——面试中最有价值的能力

</details>

### Q15. 低售罄率高毛利率的产品策略

**"你在项目四里用售罄率×毛利率双维矩阵做产品分级。面试官可能会问：如果售罄率很低但毛利率极高（比如库存很多但卖的都是高利润款）——这种产品你归到哪个象限？策略是什么？"**

<details>
<summary>点击查看回答要点</summary>

- 售罄率低 = 动销慢（滞销），毛利率高 = 利润结构好
- 不是简单的"滞销就清仓"——这种产品可能定价偏高导致转化低，但降价就损毛利
- 策略：不是直接降价，可能是捆绑搭配（买爆款送这个品试水）、或定向推给高价值用户（他们价格敏感度低）
- 这就是你学习指南里钱莹跟你讨论的内容——模型有极端情况，需要业务判断

</details>

### Q16. 补货模型中的 lead time 变化

**"你项目四的补货模型中写了 `补货点 = 预测日销 × 提前期 + 安全库存`。面试官可能会追加：如果供应商说提前期从 10 天变成了 20 天，你的模型怎么调整？安全库存应该怎么变？"**

<details>
<summary>点击查看回答要点</summary>

- 补货点 = 平均日销 × lead_time + 安全库存
- lead_time 翻倍 → 补货点上升 → 需要更早触发补货
- 但安全库存不是简单地翻倍。安全库存取决于：需求量标准差 × 服务水平对应的 z 值 × √lead_time
- lead_time 增加 → 需求的不确定性累积时间变长 → 安全库存按 √lead_time 增加
- 面试官看的是你是否知道安全库存不是线性加，而是要考虑需求波动在更长周期内的放大

</details>

## 五、简历深度追问（面试官会对着你的简历一行一行问）

### Q18. ROI 提升 15% 怎么算出来的？

**"简历上说'预算重新分配后全域 ROI 预估提升 15%+'——这个 15% 怎么算出来的？"**

<details>
<summary>点击查看回答要点</summary>

- 不是拍脑袋的。是基于预算调整后的 new_spend × 拟合的边际 ROI 曲线预计 new_revenue
- 然后用 (new_revenue / total_budget) / (old_revenue / total_budget) - 1 = 15%+
- 诚实说明这是在理想假设下的估算（假设边际 ROI 曲线在一定范围内稳定、平台竞争环境不变等）
- 面试官更看重你**知道你模型的前提假设**，而不是数字本身

</details>

### Q22. 向业务方解释统计概念

**"你要给一个不懂数据的业务方（比如黄晓明）解释'卡方检验的 p 值 < 0.05 说明什么'，你怎么用最简单的语言说清楚？"**

<details>
<summary>点击查看回答要点</summary>

- 不是讲数学。是讲"这意味着我们观察到的差异，不太可能是巧合"
- 可以类比："就像你抛硬币连续 7 次正面——可能是运气，但概率只有 0.7%——你会开始怀疑这枚硬币作弊了。同样，p < 0.05 说明'没效果纯靠巧合'的概率不到 5%"
- 从业务角度说："不同优惠券之间的转化率差异，不是随机波动，是真实存在的——你们可以放心按这个策略去发券"

</details>
